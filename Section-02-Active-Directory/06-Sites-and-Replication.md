## 📌 Overview

Most companies don't exist in a single building. They have offices in multiple cities or countries — each connected by WAN links that are slower and more expensive than local network connections.

Active Directory needs to work across all of these locations — but it needs to do so **intelligently**, without flooding slow WAN links with unnecessary traffic.

This is what **AD Sites** and **Replication** manage.

---

## 🏢 What is an AD Site?

> **Simple definition:**  
> An AD Site is a representation of a physical location (office, data centre, branch) defined by one or more IP subnets. It tells Active Directory which network is "local" (fast) and which is "remote" (slow WAN).

```
Company locations:
  London HQ     → 192.168.1.0/24  → AD Site: "London"
  Bengaluru     → 192.168.2.0/24  → AD Site: "Bengaluru"
  New York      → 192.168.3.0/24  → AD Site: "NewYork"

When Hareesh (192.168.2.x) logs in:
  AD knows he is in Bengaluru subnet
  → Directs him to the Bengaluru DC (DC-IN01) ← local, fast
  → NOT to London DC (DC01) ← slow WAN

If Bengaluru DC goes down:
  → AD falls back to London DC (over WAN)
  → Slower login, but still works ✅
```

### Why Sites Matter

```
Without Sites:
  Hareesh in Bengaluru logs in
  AD randomly picks any DC — might pick London DC
  Authentication traffic crosses the WAN for every login
  File shares, Group Policy, everything goes over slow WAN
  → Poor performance for remote users

With Sites correctly configured:
  Hareesh in Bengaluru → always uses local Bengaluru DC
  Authentication stays local → fast login ✅
  Only replication traffic crosses the WAN → controlled schedule ✅
```

## 🔗 AD Replication — Keeping All DCs in Sync

Every Domain Controller has a full copy of the AD database. When a change is made on one DC — it must be copied to ALL other DCs.

This process is called **replication**.

```
Example:
  GP resets Hareesh's password on DC01 (London)
  
  Without replication:
  → DC01 has new password
  → DC-IN01 (Bengaluru) still has old password
  → Hareesh tries to log in at Bengaluru office → old password accepted
  → or DC-IN02 (New York) still has old password → Hareesh's login fails ❌

  With replication:
  → GP resets password on DC01
  → DC01 replicates the change to DC02 (London) within seconds
  → DC01 notifies DC-IN01 (Bengaluru) → Bengaluru replicates within 15 seconds
  → Within ~15 seconds → ALL DCs have the new password ✅
```

### Intrasite vs Intersite Replication

| | Intrasite (within a site) | Intersite (between sites) |
|---|---|---|
| **Connection** | Fast LAN | Slow WAN |
| **Frequency** | Near-instant (15 second notification) | Scheduled (every 180 minutes default) |
| **Compression** | No (fast network — not needed) | Yes (conserves WAN bandwidth) |
| **Triggering** | Automatic notification | Schedule-based |

```
Intrasite (London):
  DC01 makes a change
  → Notifies DC02 after 15 seconds
  → DC02 replicates immediately ✅

Intersite (London → Bengaluru):
  DC01 makes a change
  → Change waits for the replication schedule
  → Every 3 hours (or configured interval) → replicates to DC-IN01
  → Can be configured: every 15 minutes if WAN allows it
```

## 🔄 Replication Topology — KCC

Active Directory automatically builds its own replication connections using the **KCC (Knowledge Consistency Checker)**.

```
KCC runs every 15 minutes on each DC
Evaluates: which DCs exist, which sites exist, link costs
Builds: Connection Objects (replication paths between DCs)
Ensures: no more than 3 hops between any two DCs

Example topology (auto-generated by KCC):
  London:    DC01 ←→ DC02 (bidirectional, fast)
  Bengaluru: DC-IN01 ←→ DC-IN02 (bidirectional, fast)
  
  Site link: DC01 ←→ DC-IN01 (WAN — scheduled)
             DC01 ←→ DC-US01 (WAN — scheduled)

Ring topology for >3 DCs in a site (fault tolerant):
  DC01 → DC02 → DC03 → DC01 (ring — any DC failure still allows replication)
```

## ⚙️ Configuring Sites and Subnets

### Step-by-Step Site Configuration

```
Scenario: GP is setting up AD for a new Bengaluru office

Step 1: Create the Site
  Active Directory Sites and Services
  → Right-click Sites → New Site
  → Name: Bengaluru
  → Select site link: DEFAULTIPSITELINK

Step 2: Create the Subnet
  → Expand Sites → Subnets
  → Right-click → New Subnet
  → Prefix: 192.168.2.0/24
  → Site: Bengaluru ← associates this network with Bengaluru site

Step 3: Move the DC to the correct site
  → Expand Sites → Bengaluru → Servers
  → Drag DC-IN01 to Bengaluru site

Step 4: Create a Site Link (WAN connection definition)
  → Inter-Site Transports → IP → New Site Link
  → Name: London-Bengaluru
  → Add sites: London, Bengaluru
  → Cost: 100 (higher = less preferred)
  → Replication interval: 180 minutes (or lower if WAN allows)
  → Replication schedule: business hours only (to not saturate WAN overnight)
```

### PowerShell Commands

```powershell
# View all sites
Get-ADReplicationSite -Filter *

# View all subnets
Get-ADReplicationSubnet -Filter *

# View site links
Get-ADReplicationSiteLink -Filter *

# View replication connections
Get-ADReplicationConnection -Filter *

# Check replication status between DCs
Get-ADReplicationPartnerMetadata -Target "DC01.company.local" -Scope Domain

# Force replication immediately
repadmin /syncall /AdeP

# Check replication health
repadmin /showrepl

# View replication summary (great for quick health check)
repadmin /replsummary

# Run replication diagnostics
dcdiag /test:replications
```

## ⚠️ Replication Problems — USN Rollback & Lingering Objects

### USN Rollback
```
What: A DC is restored from an old snapshot/backup
      Its USN (Update Sequence Number) goes backward in time
      Other DCs think they already have the changes → stop replicating from it

Symptom:
  Restored DC has outdated data
  New accounts created after the snapshot don't appear on the restored DC
  Password changes not replicating

Cause: Restoring a VM snapshot of a DC without using proper AD backup

Prevention:
  Never use VM snapshots to restore DCs
  Use Windows Server Backup with authoritative/non-authoritative restore
  Use VM-Generation ID aware hypervisors (Hyper-V, VMware)
```

### Lingering Objects
```
What: An object is deleted on one DC
      A DC that was offline misses the deletion
      Object stays alive on the offline DC forever (lingering)

Symptom:
  A user that was deleted still appears on some DCs
  Security groups show deleted members on some DCs

Fix:
  repadmin /removelingeringobjects DC-IN01.company.local `
            DC01.company.local "DC=company,DC=local"
```

## 🔧 Replication Troubleshooting

```powershell
# Most common replication troubleshooting commands

# 1. Check overall replication health
repadmin /replsummary
# Look for: failures, large deltas (time since last replication)

# 2. See detailed replication status
repadmin /showrepl
# Look for: "FAIL" entries, error codes

# 3. Force replication from all partners
repadmin /syncall /AdeP
# /A = all partitions, /d = identifies servers by DN, /e = enterprise, /P = push

# 4. Check specific replication error
repadmin /showrepl DC-IN01
# View what DC-IN01 is replicating from and any errors

# 5. Full DC diagnostics
dcdiag /test:replications /v

# 6. Check event logs for replication errors
Get-WinEvent -LogName "Directory Service" -ComputerName "DC01" |
    Where-Object {$_.LevelDisplayName -eq "Error"} |
    Select-Object TimeCreated, Message -First 20

# Common error codes:
# 8614 → Time difference too large (fix: sync time)
# 8456/8457 → Source/Destination DC refusing replication
# 1722 → RPC server unavailable (check firewall, DNS)
# 8461 → Replication preempted (normal, try again)
```

## 🎯 Interview Questions

**Q1. What is an AD Site and why is it important?**  
**A:** An AD Site is a representation of a physical network location defined by one or more IP subnets. It is important because it controls two things: first, it directs users to the nearest Domain Controller for authentication — users in Bengaluru authenticate against the Bengaluru DC rather than the London DC, reducing WAN traffic and improving login speed. Second, it controls replication scheduling — changes replicate immediately within a site (LAN) but follow a schedule between sites (WAN) to avoid saturating slow connections.

---

**Q2. What is AD replication and what happens if it fails?**  
**A:** AD replication is the process of copying changes made on one Domain Controller to all other DCs, keeping every DC's database consistent. If replication fails: DCs have inconsistent data — a password reset on one DC may not be known to another, causing authentication failures. New user accounts may not exist on all DCs. Group Policy changes may not apply consistently. In severe cases, a "split brain" scenario develops where DCs have diverging data. Replication health should be monitored daily using repadmin /replsummary.

---

**Q3. What is the KCC?**  
**A:** The KCC (Knowledge Consistency Checker) is an automatic process that runs on every DC every 15 minutes. It analyses the AD topology — which DCs exist, which sites exist, what site links are configured — and automatically builds replication connections between DCs. It ensures no more than 3 replication hops between any two DCs and creates redundant paths so replication continues if a DC fails. Admins can also manually create connection objects if the auto-generated topology is suboptimal.

---

**Q4. What is the difference between intrasite and intersite replication?**  
**A:** Intrasite replication happens between DCs in the same site — the same physical location connected by fast LAN. It is near-instant: a change triggers a notification after 15 seconds and the receiving DC replicates immediately. No compression is used since bandwidth is abundant. Intersite replication happens between DCs in different sites connected by WAN. It follows a configured schedule (default every 3 hours), uses compression to reduce WAN traffic, and can be restricted to certain hours to avoid saturating the WAN link during business hours.

---

**Q5. Scenario — Hareesh in the Bengaluru office reports slow logins every morning. You suspect he is authenticating against the London DC instead of the local one. How do you diagnose and fix this?**  
**A:** Diagnosis: (1) On Hareesh's machine run: nltest /dsgetdc:company.local — this shows which DC his machine found. (2) Check if 192.168.2.0/24 is correctly defined as a subnet and associated with the Bengaluru site — Get-ADReplicationSubnet -Filter *. (3) Check the Bengaluru DC is in the correct site in AD Sites and Services. Fix: (1) If subnet is missing or wrong, create/correct it: New-ADReplicationSubnet -Name "192.168.2.0/24" -Site "Bengaluru". (2) If the DC is in the wrong site, move it in AD Sites and Services. (3) Force the machine to refresh its DC selection: nltest /dsgetdc:company.local /force. After fix, Hareesh should authenticate against DC-IN01 locally.

---

*"Sites are Active Directory's understanding of your physical world. Get them right and AD works efficiently and intelligently. Get them wrong and you pay for it in slow logins, saturated WAN links, and replication failures."*
