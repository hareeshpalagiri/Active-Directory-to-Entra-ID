# Section 6.8 — B2B & B2C Identity

## 🎯 Overview

Entra ID supports two external identity scenarios:

1. **B2B (Business-to-Business)** — Partner/vendor collaboration within your org
2. **B2C (Business-to-Consumer)** — Customer identity and access management

---

## 👥 B2B Collaboration (External Partners)

### What is B2B?

**B2B** = Inviting external users (partners, vendors, contractors) to access your organization's resources.

```
┌─────────────────────────────────────────┐
│           B2B Use Cases                  │
├─────────────────────────────────────────┤
│                                         │
│ 1. Partner Collaboration                │
│    ├─ Vendors need access to files      │
│    ├─ Contractors review documents      │
│    ├─ Partners edit shared projects     │
│    └─ Example: Salesforce partner       │
│                                         │
│ 2. Consultant Access                    │
│    ├─ External consultants              │
│    ├─ Temporary project access          │
│    ├─ View-only or full access          │
│    └─ Example: Audit firm               │
│                                         │
│ 3. Guest User Invitation                │
│    ├─ One-time access                   │
│    ├─ Specific resource sharing         │
│    ├─ Time-limited                      │
│    └─ Example: Share SharePoint file    │
│                                         │
│ 4. Multi-Tenant Access                  │
│    ├─ Cross-organization scenarios      │
│    ├─ Complex federation                │
│    ├─ Mutual trust                      │
│    └─ Example: Joint venture            │
│                                         │
└─────────────────────────────────────────┘
```

### B2B Guest User Invitation

#### Method 1: Manual Invitation (Azure Portal)

```
Step 1: Entra ID → Users → + New guest user
        
Step 2: Invite external user
        Email: partner@externalcompany.com
        Personal message: "Welcome to our collaboration"
        
Step 3: Click "Invite"
        └─ Invitation email sent to partner

Step 4: Partner receives email:
        "You've been invited to Contoso"
        [Accept Invitation] button
        
Step 5: Partner clicks link
        └─ Redirects to invitation acceptance page
        
Step 6: Partner signs in:
        Option 1: With own Microsoft account
        Option 2: With own Entra ID (organization)
        Option 3: Via federation (Google, Facebook)
        
Step 7: Partner grants consent:
        "Permission: Access Contoso resources"
        [Consent] button
        
Step 8: Partner becomes guest user:
        └─ Can now access assigned resources
```

#### Method 2: PowerShell Bulk Invitation

```powershell
# Create invitation for external user
$params = @{
    invitedUserEmailAddress = "partner@externalcompany.com"
    inviteRedirectUrl = "https://myapps.microsoft.com"
    sendInvitationMessage = $true
    invitedUserMessageInfo = @{
        customizedMessageBody = "Welcome! We're excited to collaborate."
    }
}

$invitation = New-MgInvitation -BodyParameter $params

Write-Host "Invitation URL: $($invitation.InviteRedeemUrl)"

# Bulk invite from CSV
Import-Csv "partners.csv" | ForEach-Object {
    $params = @{
        invitedUserEmailAddress = $_.Email
        inviteRedirectUrl = "https://myapps.microsoft.com"
        sendInvitationMessage = $true
    }
    New-MgInvitation -BodyParameter $params
}

# CSV format:
# Email
# partner1@external.com
# partner2@external.com
```

#### Method 3: SharePoint Self-Service Sharing

```
Most common for end users:

Step 1: User shares SharePoint file/folder
        └─ Right-click → Share
        
Step 2: Enter email address:
        partner@externalcompany.com
        
Step 3: Set permissions:
        ├─ Can edit
        ├─ Can view
        └─ Expires (optional)
        
Step 4: Click "Share"
        └─ Automatic invitation sent
        
Step 5: Partner receives email
        └─ Click link to accept
        
Step 6: Partner's scope:
        ├─ Can only access shared file/folder
        ├─ Cannot see other organization content
        └─ Limited by share settings
```

### Guest User Object in Entra ID

```
When external user accepted invitation:

User Type: Guest
UPN: partner_externalcompany.com#EXT#@contoso.onmicrosoft.com
Display Name: Partner Name
Source: Invited user
Email: partner@externalcompany.com
User ID: (unique GUID)

Guest User Capabilities:
✓ Access assigned resources (SharePoint, Teams, etc)
✓ See shared files and folders
✓ Participate in Teams channels
✓ View organization directory (depends on setting)
✗ Create new resources
✗ Invite other guests (by default)
✗ See all users
✗ Create groups

Guest User Restrictions (Default):
Guest cannot:
  • See other users' full profiles
  • View internal structure
  • Access all shared resources
  • Invite others without permission

Admins can relax restrictions:
  • Allow guests to invite other guests
  • Allow guests to create groups
  • Allow guests to see more of directory
  • Set time limits on guest access
```

### Guest Access Policies

```
Configure guest access restrictions:

Entra ID → External Identities
         → External collaboration settings

Restrict who can invite guests:
├─ Only admin
├─ Admin + members
├─ Any user (default)

Guest user permissions:
├─ View limited directory (default)
├─ View full directory
├─ Can invite others

Access restrictions:
├─ Expiration (30-90 days)
├─ MFA requirement
├─ Conditional Access policy
└─ Specific resource access

Example Policy:
Guest users must:
✓ Complete MFA setup
✓ Accepted within 7 days
✓ Re-verify identity after 30 days
✓ Can only access specific apps
```

### Guest User Lifecycle

```
Timeline:

Day 0: Invitation sent
       └─ Email expires after 90 days

Day 1: Guest accepts invitation
       ├─ Creates Azure AD user account
       ├─ Consent form shown
       ├─ Added to directory
       └─ Can access assigned resources

Days 1-30: Guest actively using
           └─ Normal access

Day 30: (If policy set) Re-verify identity
        └─ Conditional Access may trigger MFA

Day 90: Access expires (if policy set)
        └─ Guest cannot sign in
        └─ Must be re-invited

Offboarding Guest:
1. Remove from groups/resources
2. Optionally delete user account
3. Deleted users: 30-day recovery window
4. After 30 days: Permanently deleted
```

### Multi-Tenant Scenarios

```
Scenario: Cross-org collaboration with Conditional Access

Organization A (contoso.com):
• Entra ID with users
• Teams, SharePoint, Outlook
• Invites guests from Organization B

Organization B (fabrikam.com):
• External org
• Invited to collaborate on project

Conditional Access for Guest Access:
Policy: "External users must use MFA"

IF:
  Guest user from fabrikam.com
  Accessing Teams channel
THEN:
  Require MFA
  Require compliant device
  
Result:
• B users get more security
• Contoso still protected
• Seamless collaboration
```

## 🛍️ B2C (Customer Identity)

### What is B2C?

**B2C** = Customer identity and access management. Customers sign up, sign in, manage profiles.

```
┌──────────────────────────────────────┐
│        B2B vs B2C Comparison        │
├──────────────────────────────────────┤
│                                      │
│ B2B (Partners):                      │
│ • Invited by company                 │
│ • Internal collaboration             │
│ • Company controls access            │
│ • Example: Share file with vendor   │
│                                      │
│ B2C (Customers):                     │
│ • Self-signup (or signup by co)     │
│ • Public-facing applications         │
│ • Customer controls own profile      │
│ • Example: E-commerce site signup   │
│                                      │
└──────────────────────────────────────┘
```

### B2C Use Cases

```
1. E-COMMERCE
   ├─ Customers create account
   ├─ Browse products
   ├─ Purchase with account
   ├─ Manage order history
   └─ Example: Online retail

2. SAAS APPLICATION
   ├─ Customers sign up
   ├─ Access cloud application
   ├─ Manage subscription
   ├─ Update payment method
   └─ Example: Project management app

3. MOBILE APP
   ├─ Users download app
   ├─ Sign up / sign in
   ├─ Save preferences
   ├─ Sync across devices
   └─ Example: Fitness tracking app

4. SOCIAL NETWORK
   ├─ Self-service signup
   ├─ Social login (Google, Facebook)
   ├─ Profile customization
   ├─ Connect with friends
   └─ Example: Community platform

5. FINANCIAL SERVICES
   ├─ Account creation
   ├─ Account verification
   ├─ Transaction history
   ├─ Security & compliance
   └─ Example: Banking app
```

### B2C Tenant Setup

```
B2C is a SEPARATE tenant from your main Entra ID:

Organization:
├─ Production Entra ID
│  ├─ Internal users
│  ├─ Employee identity
│  └─ Internal apps
│
├─ B2C Tenant (separate)
│  ├─ Customer accounts
│  ├─ External identity
│  └─ Customer apps

Why separate?
✓ Different policies for customers vs employees
✓ Different security requirements
✓ Customer data isolation
✓ Separate compliance scope
✓ Independent scaling
```

### B2C User Journeys

```
SIGN-UP FLOW:

User visits e-commerce site
        │
        ├─ "Create Account" button
        ↓
Entra ID B2C sign-up page
        │
        ├─ Enter email or phone
        ├─ Create password
        ├─ Accept terms
        ├─ Click [Sign Up]
        ↓
Email verification
        │
        ├─ OTP (one-time password) sent
        ├─ User enters OTP
        ├─ ✓ Verified
        ↓
Account created
        │
        ├─ User created in B2C tenant
        ├─ Token issued to app
        ├─ User redirected to app
        ├─ App shows "Welcome" page
        ↓
User logged in!

---

SIGN-IN FLOW:

User visits e-commerce site
        │
        ├─ "Sign In" button
        ↓
Entra ID B2C sign-in page
        │
        ├─ Enter email/phone
        ├─ Enter password
        ├─ Click [Sign In]
        ↓
Validate credentials
        │
        ├─ Check password hash
        ├─ ✓ Valid
        ↓
Issue token
        │
        ├─ Create access token
        ├─ Create refresh token
        ├─ Redirect to app
        ↓
User in app!

---

SOCIAL LOGIN:

User visits e-commerce
        │
        ├─ "Sign in with Google" button
        ↓
Google OAuth flow
        │
        ├─ Redirected to Google login
        ├─ User authorizes app
        ├─ Google returns user info
        ├─ B2C receives Google ID
        ↓
Link to B2C account
        │
        ├─ If first time: Create B2C user
        ├─ Link Google identity
        ├─ Issue token
        ↓
User logged in via Google!
```

### B2C Policies

```
User Flows (Pre-built):

1. SIGN-UP AND SIGN-IN
   ├─ Single journey for both actions
   ├─ Email or social login
   ├─ Attribute collection
   └─ Most common

2. SIGN-UP
   ├─ New account creation only
   ├─ Collect user attributes
   ├─ Email verification
   └─ Less common

3. SIGN-IN
   ├─ Existing users only
   ├─ No account creation
   ├─ Password or social
   └─ Less common

4. PASSWORD RESET
   ├─ Forgot password flow
   ├─ Email verification
   ├─ New password entry
   └─ Usually linked to sign-in

5. PROFILE EDIT
   ├─ Users update their info
   ├─ Change email, phone, name
   ├─ Attribute management
   └─ Self-service

Custom Policies:
└─ XML-based configuration
   ├─ Complete control
   ├─ Complex flows
   ├─ Conditional logic
   └─ Advanced scenarios
```

### B2C Application Registration

```
Similar to regular Entra ID:

1. Create app registration in B2C
   ├─ Name: "E-commerce Web"
   ├─ Platform: Web
   ├─ Redirect URI: https://myshop.com/callback

2. Configure permissions:
   ├─ openid
   ├─ profile
   ├─ email
   └─ offline_access (refresh token)

3. Create client secret:
   ├─ For backend to call APIs
   ├─ Store securely in app

4. Link to user flow:
   ├─ Associate app with sign-up/sign-in flow
   ├─ Flow shows on login page

5. Users sign up/in via that flow
   └─ Auto-provisioned in B2C tenant
```

### B2C Security Considerations

```
Challenges:

1. Mass Scale
   └─ Millions of customer accounts
   └─ DDoS risk
   └─ Performance scaling

2. Data Privacy
   └─ GDPR/CCPA compliance
   └─ Right to deletion
   └─ Data residency

3. Fraud Prevention
   └─ Bot account creation
   └─ Credential stuffing
   └─ MFA fatigue attacks

Best Practices:

✓ Enable MFA for sensitive operations
✓ Implement rate limiting
✓ Monitor for suspicious patterns
✓ GDPR compliance (right to be forgotten)
✓ Secure password policies
✓ Regular security audits
✓ Data encryption at rest and in transit
✓ Audit logging
```

## 🔗 B2B to B2C Comparison

| Aspect | B2B | B2C |
|--------|-----|-----|
| **Users** | Partners, vendors, contractors | Customers, end-users |
| **Tenant** | Same as main org | Separate B2C tenant |
| **Access** | Invited, controlled | Self-service, public |
| **Identity** | Organizational account | Email, phone, social |
| **Scope** | Internal resources | Public app/service |
| **Lifetime** | Project-based (months/years) | Long-term (years+) |
| **Verification** | Company email | Email/phone verification |
| **Support** | Company IT | Self-service |
| **Compliance** | Internal data policies | Customer privacy laws |

## 🎓 Interview Questions

**Q1: What's the difference between B2B and B2C in Entra ID?**
A: B2B is partner/vendor collaboration — external users invited into your org, shared resources. B2C is customer identity — self-service signup for public apps/e-commerce. B2B uses same tenant, B2C uses separate B2C tenant.

**Q2: How do you invite a guest user in B2B?**
A: Three ways: (1) Azure Portal → Users → Invite guest (2) PowerShell New-MgInvitation (3) SharePoint share button. Guest gets email invite, accepts, becomes guest user in tenant.

**Q3: What restrictions apply to guest users by default?**
A: Can access assigned resources. Cannot see other users' profiles (limited directory view). Cannot create resources. Cannot invite others (by default). Can be customized by admin policy.

**Q4: Why is B2C a separate tenant?**
A: Different policies, security requirements, compliance scope. Customers and employees have different needs. Separating scales independently, isolates data, simplifies compliance.

**Q5: How does social login work in B2C?**
A: User clicks "Sign in with Google." Redirected to Google login. Google verifies, returns user info. B2C creates/links account. User logged in to app. No password needed.
